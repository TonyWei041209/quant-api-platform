# v1.7.0 Portfolio Context Release — Implementation Plan

## 1. Why Portfolio Context

After v1.6.0 (Workspace Cohesion), the platform feels like one unified workspace.
But a critical gap remains: **the user's actual portfolio — what they hold, what
they've traded, what they're about to trade — is not naturally connected to the
research and decision-making flow.**

Currently:
- Broker readonly data (positions, orders, dividends) exists in Execution page
- Research page doesn't know if the user already holds an instrument
- Dashboard shows daily brief but not portfolio-aware context
- Watchlist, draft candidates, and current holdings are disconnected objects
- "What I hold + What I'm researching + What I plan to do" are three separate views

v1.7.0 closes this gap.

## 2. Core Theme

> **Portfolio Context Layer**: Make the user's actual holdings, recent trades,
> and execution candidates naturally visible throughout the research workflow.

## 3. Version Boundary

### Include
- Holdings awareness in Research (is this instrument in my portfolio?)
- Holdings summary in Dashboard (what do I hold today?)
- Draft/candidate vs current position visibility in Execution
- Watchlist ↔ positions cross-reference
- Portfolio-aware notes and research continuity

### Exclude
- Live trading enablement
- Complex portfolio analytics engine
- Multi-broker abstraction
- Multi-currency/multi-market portfolio
- Automated rebalancing execution
- Real-time P&L streaming

## 4. Implementation Phases

### Phase 7A: Portfolio Data Contract
- Define how positions/orders/account data flows into WorkspaceContext
- Add `portfolioPositions` and `portfolioSummary` to WorkspaceContext
- Create `/portfolio/summary` API endpoint that aggregates:
  - current positions (from broker_position_snapshot)
  - account value (from broker_account_snapshot)
  - recent orders (from broker_order_snapshot)
- Ensure data is readonly and clearly sourced from T212

### Phase 7B: Dashboard Portfolio Awareness
- Add "My Holdings" card to Dashboard
- Show top positions with current value/qty
- Show account summary (if available)
- Link positions to instrument research pages
- Show "holding this" badge on watchlist items that are also positions

### Phase 7C: Research Portfolio Context
- When viewing an instrument in Research, show:
  - "You hold X shares" badge (if in portfolio)
  - "Recent order: BUY 10 @ $xxx on date" (if recent)
  - "Active draft exists" (if draft pending)
- This is display-only, not action-changing
- Must not interfere with PIT-safe research logic

### Phase 7D: Execution Portfolio Integration
- Execution page: clearer separation of "my positions" vs "my drafts"
- Show current holding when creating intent/draft for same instrument
- Risk check enhancement: warn if draft conflicts with existing position
- "Already holding X shares" context in approval view

### Phase 7E: Cross-Object Relationships
- Watchlist items: mark which are currently held
- Notes: "note on a held position" vs "note on a candidate"
- Recent activity: include position changes from broker sync
- Continue flow: "you bought X yesterday, here's your research on it"

### Phase 7F: Docs / Tests / Release
- Update docs/execution_policy.md
- Update docs/daily-workflow.md
- Add portfolio-related tests
- Update STATUS.md, PROJECT_SUMMARY.md
- Release v1.7.0

## 5. Data Objects & Relationships

```
Portfolio Summary (from T212 readonly)
  ├── Account: value, cash, currency
  ├── Positions[]: instrument_id, ticker, qty, avg_price, current_value
  └── Recent Orders[]: instrument_id, side, qty, price, status, date

WorkspaceContext (enhanced)
  ├── activeInstrument
  ├── activeWatchlist
  ├── activePreset
  ├── recentAction
  └── portfolioPositions[]  ← NEW
      └── linked to instruments via instrument_id / ticker

Cross-references:
  - Research instrument ↔ portfolio position (held? qty? value?)
  - Watchlist item ↔ portfolio position (held?)
  - Execution draft ↔ portfolio position (conflict? overlap?)
  - Dashboard ↔ portfolio summary (total value, top holdings)
```

## 6. Key UI/UX Goals

1. User opens Dashboard → sees portfolio summary alongside daily brief
2. User opens Research on AAPL → sees "Holding 50 shares" badge
3. User opens Execution → sees current positions separate from drafts
4. User opens Watchlist → sees which items are currently held
5. User creates intent → sees warning if already holding
6. None of this requires live trading or breaks approval gates

## 7. Risks & Guardrails

- Portfolio data is readonly from T212 — must never be used to auto-execute
- Portfolio display must clearly indicate source and freshness
- "Already holding" badges must not pressure users into unwanted actions
- Must not conflate "broker says I hold X" with "platform recommends holding X"
- WorkspaceContext must not become a god object — portfolioPositions should be
  a simple array, not a complex nested state tree
- If T212 key is not configured, portfolio features gracefully degrade to
  "No broker connected" with clear guidance

## 8. Dependencies

- T212 readonly API verified (done in v1.5.x)
- broker_position_snapshot / broker_order_snapshot tables exist (done in Phase 1)
- WorkspaceContext exists (done in v1.6.0)
- PageVisibility refresh pattern exists (done in v1.6.1)
